## Dependency notes

- 001 establishes the only supported active-stack test commands and router/UI
  harnesses. All behavior-changing plans rely on it.
- 002 is deliberately mechanical and precedes the large vision diff so format
  noise does not obscure algorithm review.
- 004 establishes valid detections, landmarks, and aligned embeddings before
  005 and 006 make worker and enrollment behavior authoritative.
- 003 precedes 007 because media work must not preserve credential-bearing
  configuration or unsafe runtime permissions.
- 005 prevents a frozen capture from refreshing live state before 008 and 009
  make freshness and metrics authoritative. 008 then establishes explicit
  fresh/unknown/offline camera state before 009, 013, and 016 consume it.
- 010 establishes the Rust API/export contract before timezone and employee UI
  work extend those surfaces.
- 011 establishes validated local-calendar handling before 009 queries
  persisted daily metrics; this avoids a second timezone parser/fallback.
- 016–018 are product-direction plans. Execute them only after the P0/P1 trust
  gates they depend on are green.
- 017 follows 016 so the dashboard passes its one existing health/timezone
  value into the event ticker without a duplicate health request.
- 014 is numbered with the web-hardening work but intentionally lands after
  015–018 so shared markup is made responsive/accessibility-clean only once.

## Global boundaries

- Do not modify `apps/api/**`, invoke its test suite, or use it as an oracle.
- Preserve SQLite, flat cosine matching, Axum/Tokio/sqlx, Next.js 16,
  React 19, Tailwind 3, MediaMTX, native WebRTC, and the BMW M design language.
- Never commit biometric fixture images, embeddings derived from employees,
  credentials, tokens, or private RTSP URLs.
- No FAISS/vector database, PostgreSQL, Redis, Redux, React Query, component
  kit, Tailwind 4 migration, generic plugin layer, or media-server rewrite.

## Findings considered and rejected

- **Legacy timestamp compatibility:** excluded by the operator because the
  legacy backend is deprecated and scheduled for removal.
- **Legacy multi-camera write race and legacy test expansion:** excluded for
  the same reason.
- **FAISS/vector database:** rejected; a flat scan is clearer and sufficient
  for fewer than 50 employees.
- **PostgreSQL, Redis, queues, or microservices:** rejected; one LAN appliance
  has no demonstrated need for distributed infrastructure.
- **Redux, React Query, or a component framework:** rejected; native React
  state/fetch and existing components cover the current surfaces.
- **Tailwind 4 or a web rewrite:** rejected; neither fixes a product defect.
- **`hls.js` now:** rejected; delete the non-working fallback and add HLS only
  after a named supported browser proves WHEP cannot satisfy the requirement.
- **Playwright now:** rejected; focused Vitest/jsdom tests plus one manual media
  dress rehearsal cover current risk with much less machinery.
- **GPU/execution-provider expansion:** deferred until CPU correctness and a
  measured throughput shortfall exist.
- **Automatic MediaMTX latest-version tracking:** rejected; pin and verify one
  tested release instead.
